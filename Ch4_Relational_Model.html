Section 01

Relational Model ‚Äî Overview

üìä

What is the Relational Model?

  • Represents the database as a collection of relations
  • Informally, each relation resembles a table of values (or a flat file of records)
  • Each row in the table represents a collection of related data values
  • Rows represent real-world entities or relationships
  • Table name and column names = help interpret meaning of values
üìù

Why it's Different from a File

A relation is NOT just a table or file. It has specific characteristics that make it mathematically rigorous:

  • No duplicate tuples allowed
  • No inherent ordering of rows
  • All attribute values are atomic (single-valued)
  • NULLs allowed for unknown/missing values

Domain ‚Äî Set of Atomic Values

Definition: A domain D is a set of atomic values. By atomic we mean each value in the domain is indivisible as far as the relational model is concerned.

A common method: specify a data type or format from which the domain values are drawn. A domain is given a name, data type, and format.

Relation Schema R(A‚ÇÅ, A‚ÇÇ, ..., A‚Çô) is made up of:

‚Ä¢ The degree of a relation = number of attributes n

A relation (or relation state) r of schema R(A‚ÇÅ,...,A‚Çô), denoted r(R), is a set of n-tuples:

r = {t‚ÇÅ, t‚ÇÇ, ..., t‚Çò}

Relation State ‚Äî Dynamic Data

Relation State r(R)

  • The current relation state reflects only valid tuples that represent a particular state of the real world
  • As the real world changes ‚Üí relation is updated ‚Üí new state
  • Schema R is relatively static (rarely changes)
  • Changes only when e.g. adding a new attribute
  • r(R) changes constantly with INSERT, UPDATE, DELETE

Section 06

Ordering of Tuples ‚Äî No Order in a Set

üìê

Tuple Ordering in a Relation

  • A relation is defined as a SET of tuples
  • Mathematically, elements of a set have no order ‚Äî hence tuples have no particular order
  • Tuple ordering is NOT part of a relation definition
  • A relation represents facts at a logical/abstract level
  • The same tuples in different order = logically the same relation
üî¢

Ordering of Values Within a Tuple

The tuple itself is an ordered list of n values, so the ordering of values and attributes in the schema IS important (technically).

However, at the logical level, order of attributes doesn't really matter as long as the correspondence between attributes and values is maintained.

Alternative definition: relation r(R) is a finite set of mappings from R to D (union of all domains). In this view, ordering is completely unnecessary.

Exam Point: "Tuples in a relation have no particular order" ‚Äî this is one of the key characteristics that distinguishes a relation from a file (where records have physical order on disk).

Section 07

Characteristics of Relations

These characteristics make a relation different from a regular file or table.

Characteristic 01
No Duplicate Tuples
All tuples in a relation are distinct. No two rows can be identical. This is enforced by the Primary Key constraint.
Characteristic 02
No Ordering of Tuples
A relation is a SET of tuples ‚Äî sets have no order. Tuples do not have a "first" or "last" position in the logical definition.
Characteristic 03
No Ordering of Attributes
At the logical level, attribute order doesn't matter as long as attribute-value correspondence is maintained. Technically an n-tuple is ordered, but logically it's a mapping.
Characteristic 04
All Values are Atomic
Each value in a tuple must be an atomic (indivisible) value from its domain. Multi-valued attributes are NOT allowed ‚Äî this is 1NF (First Normal Form).
Characteristic 05
NULL Values Allowed
NULL is a special value meaning "unknown" or "not applicable." E.g., Dick Davidson has null for HomePhone ‚Äî his number is unknown. NULL ‚â† 0 or empty string.
Characteristic 06
Fixed Degree
The degree (number of attributes) of a relation is fixed for a given schema. All tuples in a relation must have exactly n values corresponding to the n attributes.

Schema R = Intension (structure/design, static). State r(R) = Extension (actual data, changes constantly). Schema changes only when adding attributes; state changes with every INSERT/UPDATE/DELETE.

Definition
NULL means "unknown" or "not applicable." E.g., Dick Davidson's HomePhone is null (unknown). NULL ‚â† 0, not empty string. A special value allowed in any domain.

Definition
Atomic = indivisible. Each value in a tuple must be a single atomic value from its domain. Multi-valued attributes NOT allowed. This is the definition of 1NF (First Normal Form).
