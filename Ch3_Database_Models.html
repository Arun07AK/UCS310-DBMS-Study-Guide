Section 02

Hierarchical Model

üå≥

Key Properties

  • Oldest data model ‚Äî developed by IBM in 1968
  • Organizes data in tree-like structure
  • Contains nodes connected by branches
  • Topmost node = root node
  • Multiple top-level nodes = root segments
  • Each node has exactly ONE parent
  • One parent may have many children
  • Implements 1:1 and 1:N relationships only
  • Pointers used to establish relationships physically
  • Examples: IMS from IBM, System 2000 from Intel
  • &XML and XAML use this model

Problems / Anomalies

Insertion Anomaly
Child node cannot be inserted without the parent node existing.
Deletion Anomaly
Difficult to delete parent node (all children would be lost).
Update Inconsistency
Update leads to inconsistency due to multiple instances of child records.
Lacks Data Independence
No data independence. Less flexible. No SQL querying support.

Section 03

Network Model

üï∏Ô∏è

Key Properties

  • Advanced version of hierarchical model
  • Uses directed graphs instead of tree structure
  • Child can have more than one parent
  • Uses concepts of Records and Sets
  • Data = collection of records; Relationships = links/pointers
  • Records organized as collections of arbitrary graphs
  • Implements 1:1, 1:N, and M:N relationships
  • CODASYL DBTG proposal ‚Äî Honeywell, UNIVAC, ICL
  • Examples: VAX-DBMS, DMS-1100 (UNIVAC), SUPRADBMS
  • Partial data independence
  • No insertion or deletion anomaly

Differences from Hierarchical

Similarity: Both use records (for data) and links/pointers (for relationships). Both are complex to design and maintain.

Section 04

Relational Model

üìä

Key Properties

  • Developed by E.F. Codd in 1970
  • No physical links (unlike hierarchical/network)
  • Data represented in table form only
  • Deals with data, not physical structure
  • Provides information regarding metadata
  • At intersection of row and column: only ONE value per tuple
  • Provides easy way to handle queries
  • Implements 1:1, 1:N, and M:N relationships
  • Logical representation: rows and columns
  • No insertion/deletion anomaly (with normalization)
  • SQL-based declarative querying supported
  • Provides full data independence
  • Easy to comprehend (physical details hidden)
  • Examples: Oracle, MySQL, SQL Server

Section 06

Flashcards

Hierarchical Model
Year + structure?
Key Facts
IBM 1968. Tree structure. Each node has EXACTLY ONE parent. 1:1 and 1:N only. Insertion/deletion anomalies. Lacks data independence. IMS from IBM.
Network Model
Key difference from hierarchical?
Key Facts
Advanced hierarchical. Directed graphs. Child can have MANY parents. Records + Sets. CODASYL DBTG. Supports M:N. No anomalies. Partial data independence.
Relational Model
Who + year?
Key Facts
E.F. Codd, 1970. Table structure. No physical links. Data only. SQL querying. Full data independence. No anomalies. Oracle, MySQL, SQL Server.
Insertion Anomaly (Hierarchical)
What's the rule?
Definition
Child node CANNOT be inserted without the parent node existing first. E.g., cannot create an account without a customer existing in the tree.
Root Node
In hierarchical model
Definition
The topmost node in a hierarchical tree. If multiple nodes appear at the top level, they are called "root segments" instead of root node.
Records & Sets
Network model concept
Definition
The two data structures used in the Network model. Records store data. Sets represent relationships (links/pointers between records). Records organized as arbitrary graphs.
CODASYL DBTG
Network model origin
Full Form
Conference on Data Systems and Languages ‚Äî Data Base Task Group. Proposed the Network model standard. Led to DDL and DML for network databases (Honeywell, UNIVAC, ICL).
Why Hierarchical Fails
3 main problems
Problems
1. Insertion anomaly (no parent = no child). 2. Deletion anomaly (parent deletion loses all children). 3. Update inconsistency (same child stored multiple times). Also: no data independence, no SQL.
Why Relational Won
Main advantages
Reasons
No physical links needed. Data independence. SQL-based querying. Easy for users (physical details hidden). No anomalies with normalization. M:N relationships supported. Used by Oracle, MySQL.

Section 08

Cheat Sheet

Hierarchical
IBM 1968. Tree. ONE parent per node. 1:1 & 1:N. IMS/System2000. XML/XAML.
Network
CODASYL DBTG. Graph. MANY parents. Records+Sets. 1:1, 1:N, M:N. VAX-DBMS, DMS-1100.
Relational
E.F. Codd 1970. Tables. No physical links. SQL. Oracle, MySQL. Full data independence.
Hierarchical Anomalies
Insertion (no parent=no child), Deletion (parent loss=children loss), Update inconsistency.
Data Independence
Hierarchical: NONE. Network: PARTIAL. Relational: FULL.
Querying
Hierarchical: None. Network: None. Relational: SQL-based declarative querying.
Relationships
Hierarchical: 1:1, 1:N. Network & Relational: also M:N.
Flexibility
Hierarchical: least flexible. Network: more flexible. Relational: most flexible.
Implementation
Hierarchical: pointers. Network: linked list. Relational: rows and columns (logical).
