Section 01

Concept of Null or Unknown Value

What is NULL?

  • NULL represents a value for an attribute that is currently unknown or it is not applicable for this tuple
  • NULL is NOT the same as 0 or zero value or a text string
  • It represents absence of value

Two meanings of NULL:
‚Ä¢ Unknown ‚Äî the value exists but we don't know it (e.g., Rajesh's age is unknown)
‚Ä¢ Not Applicable ‚Äî the attribute doesn't apply to this entity (e.g., Amit has no pension)

Section 02

Why Do We Need Constraints?

Relational Integrity Constraints are referred to as conditions which must be present for a valid relation. Derived from the rules the database represents.

Issues in Basic STUDENT Table (without constraints)

A basic table suffers from following anomalies:

‚ùå No check on storing duplicate records
‚ùå No mandatory column (can skip required fields)
‚ùå No check on validity of data (any value accepted)

Solution: To ensure all these features, there is a need to apply constraints during creation of table.

Section 03

Relational Integrity Rules

RDBMS constraints are divided into three main categories:

1. Domain Constraints

Enforces valid entries for a given column by restricting the type, format, or range of values. E.g., Age must be integer between 0‚Äì150.

2. Entity Integrity (Key)

No duplicate rows in a table. Primary key must be unique and NOT NULL. If composite PK, none of its fields can be NULL.

3. Referential Integrity

Rows cannot be deleted which are used by other records (via FK). FK value must match existing PK in the referenced table.

4. User-Defined Integrity

Enforces specific business rules that do not fall into entity, domain, or referential integrity categories.

Levels of Constraints

Column Level Constraints

Constraints applied on a single column of a table.

Example: Roll_Number is Unique

Table Level Constraints

Constraints applied on more than one column of a table.

Example: Roll_Number and Class combination is unique (in primary school, roll numbers restart per class)

Section 04

NOT NULL Constraint

NOT NULL
NOT NULL
  • By default, a column can hold NULL values
  • NOT NULL specifies that all rows must have a value for that column
  • If you don't want NULL in a column ‚Üí define NOT NULL constraint
  • NULL represents unknown data ‚Äî not same as no data or 0

Syntax:

Columnname datatype(size) [CONSTRAINT constraintname] NOT NULL

Column Level Example:

CREATE TABLE student ( Roll_Number Number(4) NOT NULL, Name Char(15), Class Char(10), Marks Number(4), DOB Date );

CUSTOMERS Example:

CREATE TABLE CUSTOMERS ( ID INT NOT NULL, NAME VARCHAR(20) NOT NULL, AGE INT NOT NULL, ADDRESS CHAR(25), SALARY DECIMAL(18,2), PRIMARY KEY(ID) );

Add NOT NULL to existing table:

ALTER TABLE CUSTOMERS MODIFY SALARY DECIMAL(18,2) NOT NULL;
Section 05

UNIQUE Constraint

UNIQUE
UNIQUE
  • Prevents two records from having identical values in a column
  • Enforces uniqueness of a column or group of columns
  • Example: Prevent two customers from having the same age
  • Unlike PRIMARY KEY, UNIQUE allows NULL values

Syntax:

Columnname datatype(size) [CONSTRAINT constraintname] UNIQUE

Column Level:

CREATE TABLE student ( Roll_Number Number(4) UNIQUE, -- or with name: Roll_Number Number(4) CONSTRAINT RNO_Unique UNIQUE, ... );

Table Level (multi-column unique):

CREATE TABLE primary_student ( Roll_Number Number(4), Class Char(10), ... CONSTRAINT RNO_Class_Unique UNIQUE(Roll_Number, Class) );

Add/Drop UNIQUE:

ALTER TABLE CUSTOMERS ADD CONSTRAINT myUnique UNIQUE(AGE, SALARY); ALTER TABLE CUSTOMERS DROP CONSTRAINT myUnique;
Section 06

PRIMARY KEY Constraint (Entity Integrity)

PRIMARY KEY
PRIMARY KEY = UNIQUE + NOT NULL

Entity Integrity (Primary Key) Constraints:

  • Combination of UNIQUE + NOT NULL
  • Used to identify each row uniquely
  • May be a single field (Simple PK) or group of fields (Composite PK)
  • Primary key cannot have NULL value
  • If composite PK ‚Üí NONE of the fields can be NULL
  • Oracle enforces PRIMARY KEY using indexes

Syntax:

Columnname datatype(size) [CONSTRAINT name] PRIMARY KEY

Column Level PK:

CREATE TABLE student ( Roll_Number Number(4) PRIMARY KEY, -- or with constraint name: Roll_Number Number(4) CONSTRAINT RNO_PK PRIMARY KEY, Name Char(15), ... );

Table Level (Composite PK):

CREATE TABLE primary_student ( Roll_Number Number(4), Class Char(10), ... CONSTRAINT RNO_Class_PK PRIMARY KEY(Roll_Number, Class) );

Note: CustomerID=1 is PK for "Google". Only ONE Google can have CustomerID=1.

Section 07

CHECK Constraint

CHECK
CHECK Constraint
  • Allows Oracle to verify the validity of data being entered
  • CHECK followed by parenthesized conditions
  • Must be a logical expression evaluating to TRUE or FALSE
  • Applies check constraint only for stable values (not for values that change in future)

Limitation: Not suitable if check values need to change in future. E.g., if new class "MBA" is added, you'd need to drop and recreate. Best for marks‚â•0, gender (male/female/transgender).

Syntax:

Columnname datatype(size) [CONSTRAINT name] CHECK(logical expression)

Column Level:

CREATE TABLE student ( Roll_Number Number(4), Class Char(10) CONSTRAINT class_check CHECK(class IN('BE','ME','MCA')), Marks Number(4) CHECK(marks>=0), DOB Date );

Table Level (cross-column):

CREATE TABLE library ( Roll_Number Number(4) PRIMARY KEY, Book_Number Number(10) NOT NULL, DOI Date, DOR Date, CONSTRAINT Date_Check CHECK(DOR > DOI) );

CUSTOMERS (AGE‚â•18):

AGE INT NOT NULL CHECK(AGE >= 18), -- Alter: ALTER TABLE CUSTOMERS MODIFY AGE INT NOT NULL CHECK(AGE>=18); -- Drop: ALTER TABLE CUSTOMERS DROP CONSTRAINT myCheckConstraint;
Section 08

FOREIGN KEY Constraint ‚Äî Referential Integrity

FOREIGN KEY
FOREIGN KEY ‚Äî Referential Integrity Constraint
  • Based on concept of Foreign Keys
  • FK refers to a key attribute that should be referred to in other relationships
  • FK value in child table must match existing PK in parent table
  • Table with FK = Child Table; Table it references = Parent/Master Table
  • FK may be single column or combination of columns
  • Helps avoid data input errors, inconsistency, and quality problems
  • Cannot change PK that has matching child table records
  • Cannot delete parent record if FK references it

Syntax:

Columnname datatype(size) REFERENCES tablename[(columnname)] [ON DELETE RESTRICT|CASCADE|SET NULL]

EMP-DEPT Case Study:

-- 1. Create parent table first: CREATE TABLE department ( deptno number(2) PRIMARY KEY, dname char(20) ); -- 2. Create child table with FK: CREATE TABLE employee ( eno number(2) PRIMARY KEY, ename char(20), job char(10), deptno number(2) REFERENCES department(deptno) );

ON DELETE Options for Foreign Key

When you try to delete deptno=30 from Department table (but Surinder in Employee references it):

ON DELETE RESTRICT

Default behavior. Deletion of parent record is NOT ALLOWED if child records reference it. Dept 30 cannot be deleted while Surinder exists.

deptno number(2) REFERENCES department(deptno) ON DELETE RESTRICT;
ON DELETE CASCADE

Delete parent ‚Üí automatically deletes all child records too. Deleting dept 30 also fires Surinder. Both removed together.

deptno number(2) REFERENCES department(deptno) ON DELETE CASCADE;
ON DELETE SET NULL

Delete parent ‚Üí child records remain but FK set to NULL. Dept 30 deleted, Surinder stays but his Deptno becomes NULL.

deptno number(2) REFERENCES department(deptno) ON DELETE SET NULL;

Section 09

DEFAULT Constraint

DEFAULT
DEFAULT Constraint

The DEFAULT constraint provides a default value to a column when the INSERT INTO statement does not provide a specific value.

  • Allows inserting rows without specifying every column value
  • Not actually a constraint ‚Äî used to insert default value if value is missing
  • The datatype of the default value should match the column's datatype

Syntax:

Columnname datatype(size) DEFAULT value

Column Level (student):

CREATE TABLE student ( Roll_Number Number(4) PRIMARY KEY, Name Char(15), Class Char(10) CHECK(class IN('BE','ME','MCA')), Marks Number(4) DEFAULT 60, DOB Date );

CUSTOMERS (SALARY = 5000.00 by default):

CREATE TABLE CUSTOMERS ( ... SALARY DECIMAL(18,2) DEFAULT 5000.00, ... );

Add/Drop DEFAULT:

-- Add: ALTER TABLE CUSTOMERS MODIFY SALARY DECIMAL(18,2) DEFAULT 5000.00; -- Drop (set back to NULL): ALTER TABLE CUSTOMERS MODIFY column_name DEFAULT NULL;
Section 10

Managing Constraints with ALTER TABLE

ALTER TABLE ‚Äî What It Can Do

Add or Drop Constraint
Can ADD or DROP a constraint, but NOT modify its structure
Enable or Disable
Can enable or disable existing constraints
Add NOT NULL
Add NOT NULL using the MODIFY clause

General ALTER TABLE Syntax:

ALTER TABLE table ADD [CONSTRAINT constraint_name] type (column);

Example ‚Äî Adding FK Constraint:

ALTER TABLE employees ADD CONSTRAINT emp_manager_fk FOREIGN KEY(manager_id) REFERENCES employees(employee_id);

This adds a self-referencing FK ‚Äî a manager must already exist as a valid employee.

Viewing Constraints ‚Äî USER_CONSTRAINTS Table

Oracle maintains the USER_CONSTRAINTS table internally. You can query it to see all constraint definitions.

SELECT * FROM USER_CONSTRAINTS WHERE TABLE_NAME = 'STUDENT'; SELECT constraint_name, constraint_type, search_condition FROM user_constraints WHERE table_name = 'EMPLOYEES';

To view columns associated with constraints:

SELECT constraint_name, column_name FROM user_cons_columns WHERE table_name = 'EMPLOYEES';

Constraint type codes: C = Check/Not Null, P = Primary Key, U = Unique, R = Referential (FK)

Section 11

Flashcards

NOT NULL
Default behavior?
Definition
By default columns CAN be NULL. NOT NULL forces every row to have a value. NULL represents unknown data, not 0 or empty. Syntax: columnname datatype NOT NULL.
UNIQUE
Difference from PK?
Definition
Prevents identical values in a column/group of columns. Unlike PK, UNIQUE ALLOWS NULL values. A table can have multiple UNIQUE constraints but only ONE PK.
PRIMARY KEY
= which two?
Definition
= UNIQUE + NOT NULL. Uniquely identifies every row. Cannot be NULL. Only ONE per table. Simple or Composite PK. For composite PK, none of the fields can be NULL. Oracle uses indexes.
CHECK Constraint
What does it evaluate?
Definition
Verifies validity of data. CHECK(logical expression) ‚Äî evaluates to TRUE or FALSE. Example: CHECK(age >= 18), CHECK(class IN ('BE','ME','MCA')). Apply only for stable values.
FOREIGN KEY
Child vs Parent table
Definition
Enforces referential integrity. FK value must match existing PK in parent table. Table with FK = Child; referenced table = Parent/Master. ON DELETE: RESTRICT/CASCADE/SET NULL.
DEFAULT Constraint
When is it used?
Definition
Provides default value when INSERT doesn't specify one. Not actually a constraint. Datatype must match column. Syntax: columnname datatype DEFAULT value. E.g., SALARY DEFAULT 5000.00.
ON DELETE CASCADE
What happens to child?
Effect
When parent record is deleted, ALL corresponding child records are ALSO deleted automatically. E.g., deleting dept 30 also fires Surinder from employee table.
ON DELETE SET NULL
What happens to child?
Effect
When parent record is deleted, child records REMAIN but their FK column is set to NULL. E.g., deleting dept 30 keeps Surinder in employee table but his Deptno becomes NULL.
Entity Integrity
Which constraint?
Definition
No duplicate rows in a table. Enforced by PRIMARY KEY constraint. PK must be unique and NOT NULL. If composite PK, none of its component fields can be NULL.
Referential Integrity
Which constraint?
Definition
Enforced by FOREIGN KEY. FK value in child table must match existing PK in parent table. Cannot delete parent record that has matching child records (unless CASCADE/SET NULL used).
Domain Integrity
Which constraints?
Definition
Enforced by CHECK, NOT NULL, DEFAULT, UNIQUE. Ensures valid entries for a column by restricting type, format, or range of values.
USER_CONSTRAINTS
What is it?
Definition
Oracle's internal table that maintains all constraint definitions. Query it: SELECT * FROM USER_CONSTRAINTS WHERE TABLE_NAME='tablename'. For columns: USER_CONS_COLUMNS.

Cheat Sheet

NULL
Unknown or Not Applicable. NOT 0 or empty. Represents absence of value.
3 Integrity Rules
Domain (valid type/range), Entity (no duplicates, PK), Referential (FK matches PK).
NOT NULL
Default allows NULL. NOT NULL forces value in column. Alter: MODIFY col datatype NOT NULL.
UNIQUE
No identical values. Allows NULL. Multiple per table. Col or table level. Add/Drop with ALTER TABLE.
PRIMARY KEY
= UNIQUE + NOT NULL. Only 1 per table. No NULL. Simple or Composite. Oracle uses indexes.
CHECK
Logical expression (TRUE/FALSE). Not for values that may change. Good for marks‚â•0, gender.
FOREIGN KEY
Child references Parent PK. Enforces referential integrity. Multiple per table. NULL allowed.
DEFAULT
Not actually a constraint. Provides default value when INSERT omits column value. Datatype must match.
ON DELETE Options
RESTRICT (default, block delete), CASCADE (delete child too), SET NULL (keep child, FK=NULL).
Levels
Column level = single column. Table level = multiple columns together.
ALTER TABLE
ADD CONSTRAINT / DROP CONSTRAINT / MODIFY. Can enable/disable constraints.
USER_CONSTRAINTS
Oracle internal table. Query to see all constraints. USER_CONS_COLUMNS for columns.
