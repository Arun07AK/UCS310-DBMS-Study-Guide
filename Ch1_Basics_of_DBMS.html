Data vs. Information

Understanding the difference between Data and Information is the foundation of DBMS. The term "Database" requires understanding of both.

Data is stored facts.

Data may be:
‚Ä¢ Numerical data ‚Äî integers or floating point numbers
‚Ä¢ Non-numerical data ‚Äî characters, date, etc.

Example: 98, 89, 87, 92 are data (could be distance, marks, rupees ‚Äî we don't know!)

What is a Database?

Definition: A database is a computer-based record-keeping system whose overall purpose is to record and maintain information.

Also defined as: A collection of information organised in such a way that a computer program can quickly select desired pieces of data.

Key insight: Related information, when placed in an organised form, makes a database. Organisation of data is NECESSARY because unorganised information has no meaning.

Section 03

What is DBMS?

DBMS (DataBase Management System) ‚Äî A collection of programs that enables you to store, modify, and extract information from a database.

The general purpose of a DBMS is to provide for the definition, storage, and management of data that can be shared by many users.

A DBMS is the software system that allows users to define, create and maintain a database and provides controlled access to the data.

E.g. Oracle's DBMS is patterned on the relational model described by Dr. E. F. Codd.

Role of DBMS in an Organisation

A corporate database serves all levels of the organisation:

Strategic Data ‚Üí Policies, Goals & Strategic Planning
Managerial Data ‚Üí Management Planning & Control
Operational Data ‚Üí Operational Activities & Management
üåé Real World ‚Üí Corporate Database

The real world feeds into the Corporate Database, which serves all three management levels

Section 05

Evolution of DBMS

DBMS evolved to "close the gap" between real-world needs and computer capabilities. Here's the journey:

1950s
File Systems
Sequential records, flat files. ISAM (Indexed Sequential Access Method) and VSAM (Virtual Storage Access Method). Basic operations: Open, Close, Read, Write, Delete. Programmers used C, C++ to access files directly.
1960s ‚Äì 1970s
Hierarchical & Network DB Systems
Hierarchical model: tree structure (parent-child). Network model: graph structure (many-to-many). Better than flat files but complex and rigid.
1980s
Relational DB Systems
Proposed by Dr. E. F. Codd. Data stored in tables (relations). SQL introduced. Oracle, MySQL, SQL Server emerged. Most widely used today.
1980s ‚Äì Today
Object-Oriented DB Systems
Combines OOP concepts with database capabilities. Supports complex data types, inheritance, encapsulation. Used in engineering, multimedia applications.

File-Based Approach

File-based systems were an early attempt to computerize the manual filing system. Understanding their problems explains WHY DBMS was invented.

üìÅ

How File Systems Worked

  • Used storage devices like hard disk or CD-ROM
  • Maintained physical location of files
  • Programmers used C, C++ to directly access flat files
  • Application programs performed data management
üóÇÔ∏è

File Systems (1950s)

  • Basic constructs: Sequential records, sequential fields
  • Relied on indexes for random access
  • ISAM ‚Äî Indexed Sequential Access Method
  • VSAM ‚Äî Virtual Storage Access Method
  • Basic operations: Open, Close, Reset, Read, Write, Delete

Section 07

Limitations of File-Based Approach

These limitations are why DBMS was invented. Every single point is exam-worthy!

01
Separated & Isolated Data
Data is scattered across multiple files. Hard to combine data from different files for a single query.
02
Duplication of Data
Same data stored in multiple files (e.g., student address in both student file and exam file). Wastes space.
03
Data Dependence
Application programs depend on physical structure of files. Change file format ‚Üí must rewrite all programs.
04
Difficulty in User's View
Hard to show data from user's perspective. Different users need different views of same data.
05
Data Integrity Issues
No way to enforce business rules (e.g., age must be > 0). Invalid data can be stored easily.
06
Incompatible File Formats
Files created by different programs have different formats. Hard to share data between applications.
07
Data Security Issues
Difficult to control who can access which data. No fine-grained access control.
08
Uncontrolled Redundancy
Data redundancy leads to data inconsistency ‚Äî same data has different values in different files.
09
Poor Data Sharing
Multiple users cannot efficiently share the same data simultaneously.
10
Low Productivity
Writing and maintaining separate programs for each file operation is time-consuming and costly.
11
High Maintenance Cost
Record format vs user requirements and programs vs record format creates constant maintenance burden.

Data Consistency vs. Data Integrity

Data Consistency

Data is uniform across all systems. When data is changed at one site, it is automatically propagated to all other sites so that every copy stays identical.

Example: If a student's address is updated in the General Office file, the same change must reflect in Library, Hostel, and Account Office files. In file-based systems, this does NOT happen ‚Äî leading to inconsistency.

Data Integrity

Data is accurate, reliable, and error-free throughout its entire lifecycle. Integrity constraints (NOT NULL, CHECK, UNIQUE, FK) ensure that only valid data enters the database.

Example: A CHECK constraint ensures age > 0 and age < 150. A NOT NULL constraint ensures every student has a name. Without DBMS, invalid data can be stored freely.

Section 09

Components of DBMS Environment

A DBMS environment consists of five essential components that work together to manage data effectively.

üñ•Ô∏è

Hardware

The physical devices ‚Äî computers, storage devices (hard disks, SSDs), networking devices, I/O devices. Ranges from a single PC to a network of computers. The DBMS and applications run on this hardware.

üíø

Software

The DBMS software itself, the operating system, network software, and application programs. The DBMS is the most important software component ‚Äî it sits between the physical database and the users.

üìä

Data

The most important component. Includes both the actual data (operational data) and the metadata (data about data ‚Äî schema, constraints, descriptions). Data acts as a bridge between hardware/software and users.

üë§

Users

People who interact with the database. Includes DBA (Database Administrator), application programmers, sophisticated (casual) users, and naive (parametric) end users. Each type has different access levels.

üìã

Procedures

The rules and instructions that govern the design and use of the database. Includes procedures for logging in, using the DBMS, taking backups, generating reports, and handling failures.

Section 10

Disadvantages of DBMS

While DBMS offers many advantages, it also comes with certain drawbacks that must be considered.

01
Complexity
DBMS is complex software. Users, designers, and administrators must understand its functionality to take full advantage of it. Misuse can cause serious damage.
02
Size
The complexity and breadth of functionality makes DBMS an extremely large piece of software, occupying many megabytes of disk space and requiring substantial memory to run efficiently.
03
Performance
A file-based system is written for a specific application and may perform faster for that task. DBMS is general-purpose, so it may not run as fast as a dedicated custom solution.
04
Higher Impact of a Failure
Since all users and applications rely on the central DBMS, a failure can bring the entire system to a halt, affecting all operations simultaneously.
05
Cost of DBMS
Commercial DBMS software (Oracle, SQL Server) can be very expensive. Additional costs include licensing, annual maintenance fees, and vendor support contracts.
06
Additional Hardware Costs
DBMS requires more disk storage and memory to run. May need to upgrade hardware or buy additional servers to meet performance requirements.
07
Cost of Conversion
Migrating from a file-based system to DBMS involves significant cost ‚Äî training staff, hiring specialists, and converting existing data and applications.
